# Reference Abstraction Rule

## 검색 키워드

reference abstraction, similar game, core fun, core verb, feedback loop, prototype slice, game reference analysis

## 목적

사용자가 특정 게임, 앱, 서비스, 작품을 레퍼런스로 말했을 때, 그 작품을 그대로 복제하지 않고 프로토타입 가능한 핵심 요소로 추상화한다.

레퍼런스는 기능 목록이 아니라 "왜 재미있는가"를 찾기 위한 단서다. 이름 있는 상용 게임을 언급했더라도 초기 프로토타입은 전체 장르를 복제하지 않고 핵심 긴장/선택/피드백 루프만 검증한다.

## 분석 항목

1. Camera / View
   - 시점, 카메라 거리, 조작 방식
2. Core Verb
   - 사용자가 반복해서 하는 핵심 행동
   - 예: 공격, 회피, 조준, 점프, 탐색, 건설, 선택, 대화, 거래
3. Interaction Target
   - 핵심 행동의 대상
   - 예: 적, 플랫폼, 자원, 카드, NPC, 퍼즐 오브젝트, 건물, 유닛
4. Feedback Loop
   - 손맛, 타격감, 보상, 반응, 점수, 성장, 해금
5. State Loop
   - 주요 상태 전이
   - 예: Idle -> Attack -> Recovery -> Hit -> Stagger
6. Tension / Decision Loop
   - 플레이어가 어떤 압박 속에서 어떤 결정을 반복하는가?
   - 예: 자원 부족, 시간 제한, 안전 지역 압박, 적 시야, 카드 비용, 위험/보상 선택
7. Prototype Slice
   - 1~2주 안에 검증할 최소 장면

## 레퍼런스 압축 규칙

레퍼런스가 큰 게임이면 다음 문장으로 먼저 압축한다.

```text
이 레퍼런스의 초기 프로토타입 핵심은 [기능 목록]이 아니라,
[핵심 압박] 속에서 [핵심 행동]을 선택하고 [즉각 피드백]을 받는 루프다.
```

예:

- 배틀로얄: 대규모 멀티플레이 전체가 아니라, 제한된 자원 + 안전지대 압박 + 교전/회피 선택을 검증한다.
- 소울라이크: 보스전 전체가 아니라, 적 공격 읽기 + 회피/반격 + 피격 피드백을 검증한다.
- 덱빌딩: 전체 런 구조가 아니라, 드로우 + 비용 지불 + 카드 효과 해결 + 턴 종료를 검증한다.

## 금지

- 레퍼런스 게임의 모든 시스템을 한 번에 구현하지 않는다.
- 성장, 장비, 스토리, 보스전, 저장, 상점, 스킬트리 같은 상용 게임 주변 시스템을 초기 프로토타입에 넣지 않는다.
- 레퍼런스의 고유명사나 세부 시스템을 범용 규칙처럼 쓰지 않는다.
- 레퍼런스의 온라인 규모, 매치메이킹, 랭킹, 라이브 서비스 요소를 첫 프로토타입의 Must Have로 넣지 않는다.
