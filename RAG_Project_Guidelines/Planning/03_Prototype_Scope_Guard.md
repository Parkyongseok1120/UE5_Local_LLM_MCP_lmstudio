# Prototype Scope Guard

## 검색 키워드

prototype scope, MVP, minimum prototype, scope control, avoid overengineering, planning no multiplayer, first slice

## 목적

초기 프로토타입은 전체 게임이 아니라 핵심 가설 검증이다. 범위가 커지는 순간 핵심 재미 검증이 늦어진다.

## 기본 제한

초기 프로토타입은 기본적으로 다음 범위 안에서 시작한다.

- 플레이어 또는 주요 조작 대상 1개
- 적 또는 상호작용 대상 1~2종
- 테스트 맵/스테이지 1개
- 핵심 행동 1~3개
- 핵심 자원/아이템/무기/카드 1~3개
- UI 최소화
- 저장/성장/상점/장비/스킬트리/멀티플레이는 초기 제외

## Must Have 상한

Must Have는 핵심 가설 검증에 꼭 필요한 것만 둔다.

- 기능 수는 기본 3~5개로 제한한다.
- "레퍼런스에 있으니까"가 아니라 "첫 플레이 루프 검증에 없으면 안 되니까"를 기준으로 넣는다.
- 장르 정체성상 중요해도 첫 검증에 필요 없으면 Should Have 또는 Later로 보낸다.

예:

- 배틀로얄의 Multiplayer는 최종 제품에는 핵심이지만, 첫 프로토타입에서는 봇/더미 타겟/로컬 시뮬레이션으로 생존 압박을 먼저 검증할 수 있다.
- 슈터의 무기 다수는 나중 문제다. 첫 검증은 무기 1종과 명중 피드백이면 충분하다.
- 액션 전투의 보스 패턴 다수는 나중 문제다. 첫 검증은 공격 1개, 회피 1개, 피격 반응이면 충분하다.

## 제외 우선순위

사용자가 명시적으로 요구하지 않으면 다음을 나중으로 미룬다.

- Multiplayer / Replication / RPC
- GAS
- 대형 Manager 구조
- 저장/로드
- 장비/스킬트리/상점
- 복잡한 AI Director
- 다수의 적/무기/카드/자원
- 완성형 UI/UX

## 첫 구현 단위

첫 구현 단위는 "핵심 루프를 30초 안에 반복해볼 수 있는 테스트 씬"이어야 한다. 아트, 콘텐츠, 확장 시스템보다 조작, 반응, 피드백이 먼저다.

첫 구현 단위에 포함할 수 있는 것:

- 플레이어 조작 1개 묶음
- 상호작용/적/타겟 1종
- 핵심 피드백 1종
- 실패 또는 성공 조건 1개

첫 구현 단위에 넣지 않는 것:

- 전체 매치 구조
- 라이브 서비스 요소
- 장기 성장
- 여러 맵/여러 캐릭터/여러 무기 대량 추가
